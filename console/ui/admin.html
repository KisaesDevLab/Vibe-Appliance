Help & setup guides

      
Reference for connecting this appliance to the internet via different paths. Pick the one that matches your network situation. Click a title to expand.


      

        Custom DNS — own domain via public DNS
        

          
When to use: Your appliance has a public IP, and you own a domain. Standard public-internet HTTPS via Let’s Encrypt; Caddy auto-issues and renews certs.


          
Prerequisites

          
    
            
  • Domain registered (any registrar)
    • 
            
  • Public-facing host with ports 80 and 443 reachable from the internet
    • 
            
  • An email address for Let’s Encrypt expiry notifications
    • 
          


          
Steps

          
    
            
  1. At your DNS provider, create A records:
              
    yourdomain.com       A   <your-public-ip>
*.yourdomain.com     A   <your-public-ip>
    
            
    2. 
            
  2. Verify propagation from a different machine:
              
    dig +short tb.yourdomain.com
# should return your public IP
    
            
    3. 
            
  3. Open ports 80 and 443:
              
    sudo ufw allow 80,443/tcp
    
              In the DigitalOcean control panel, also confirm any cloud-firewall rules allow 80/443 from 0.0.0.0/0.
            
    4. 
            
  4. Switch the appliance to domain mode:
              
    sudo /opt/vibe/appliance/bootstrap.sh \
  --mode domain \
  --domain yourdomain.com \
  --email you@yourdomain.com
    
            
    5. 
          


          
Verification

          
    
            
  • https://yourdomain.com/admin loads with a green padlock
    • 
            
  • First request to each new subdomain takes 10–30s while Caddy issues its cert — subsequent requests are instant
    • 
          


          
Gotchas

          
    
            
  • Port 80 must be reachable from the public internet for ACME HTTP-01. If port 80 is blocked, certs never issue and every subdomain returns broken — use Cloudflare Tunnel or DNS-01 wildcard certs instead.
    • 
            
  • DNS propagation can take minutes to hours. Don’t switch to domain mode until dig returns the right IP from a fresh machine, or you’ll waste cert-issuance attempts (Let’s Encrypt rate-limits at 5 failures per hostname per hour).
    • 
            
  • Switching from LAN/Tailscale mode rewrites your URLs — update bookmarks and any app callback URLs.
    • 
          

        

      


      

        Tailscale — private mesh, no public IP needed
        

          
When to use: Private deployment — only your team accesses the appliance, no public exposure needed. Or your host has no public IP at all.


          
Prerequisites

          
    
            
  • Tailscale account (free for up to 100 devices)
    • 
            
  • Auth key from your Tailscale admin console
    • 
          


          
Steps

          
    
            
  1. Generate an auth key at https://login.tailscale.com/admin/settings/keys. Recommended: reusable, no expiry — otherwise the appliance will go offline silently when the key expires.
    2. 
            
  2. Switch the appliance to Tailscale mode:
              
    sudo /opt/vibe/appliance/bootstrap.sh \
  --mode tailscale \
  --tailscale-authkey tskey-auth-...
    
            
    3. 
            
  3. Wait a minute or two for the tailnet hostname to propagate. Verify on the host:
              
    tailscale status
    
            
    4. 
          


          
Verification

          
    
            
  • From any tailnet member: https://<hostname>.<your-tailnet>.ts.net/admin loads with a valid Tailscale-CA cert
    • 
            
  • Cockpit: https://<hostname>.<your-tailnet>.ts.net:9090
    • 
          


          
Gotchas

          
    
            
  • Every user accessing the appliance needs Tailscale on their own device.
    • 
            
  • Tailscale’s MagicDNS auto-issues certs from Tailscale’s own CA — no Let’s Encrypt, no port 80 requirement.
    • 
            
  • Vibe-Connect’s client-portal magic-links won’t work if your CLIENTS aren’t on Tailscale (they almost certainly aren’t). For external client traffic, you need Custom DNS or Cloudflare Tunnel; Tailscale is staff-only.
    • 
          

        

      


      

        Cloudflare Tunnel — no port forwarding, no public IP
        

          
When to use: Public clients need to reach the appliance, but you can’t (or won’t) open ports 80/443. Common cases: home installs behind double-NAT or carrier-grade NAT, residential ISPs that block inbound TCP/80, datacenter VMs on internal networks, or adding Cloudflare’s DDoS protection without changing infrastructure.


          

            Easiest path: use the in-page wizard. Open
            Configuration → Network, toggle
            Cloudflare Tunnel ON, and the four-step wizard appears
            below the form. Paste an API token; the wizard verifies it,
            auto-detects your account & zone IDs, programmatically writes
            the four fields, and provisions the tunnel via an admin endpoint
            — no SSH required. Steps below are the manual / power-user
            path; both produce the same end-state.
          


          
Prerequisites

          
    
            
  • Cloudflare account (free).
    • 
            
  • Your domain’s DNS hosted on Cloudflare’s nameservers. The domain’s registration can stay anywhere (Namecheap, GoDaddy, etc.) — only the NS records change. The switch is free, doesn’t touch registration, and propagates in minutes.
    • 
            
  • Outbound TCP 7844 allowed from the host (virtually all ISPs and firewalls allow this).
    • 
          


          
Steps (manual path — skip if using the wizard above)

          
    
            
  1. Move DNS to Cloudflare if you haven’t already. Cloudflare’s onboarding flow:
              
      
                
    • Sign up at cloudflare.com, click Add site, enter your domain.
      • 
                
    • Cloudflare imports your existing DNS records and gives you two nameservers.
      • 
                
    • At your registrar (e.g. Namecheap → Domain List → Manage → Domain tab → Nameservers), set Custom DNS and paste Cloudflare’s two nameservers.
      • 
                
    • Wait for propagation: dig NS yourdomain.com +short should show Cloudflare’s nameservers.
      • 
              
    
            
    2. 

            
  2. Create a Cloudflare API token at dash.cloudflare.com/profile/api-tokens → Create Token → Custom token. Set permissions:
              
      
                
    • Account → Cloudflare Tunnel → Edit (lets the script create + manage the tunnel)
      • 
                
    • Zone → DNS → Edit (lets the script create CNAMEs for each subdomain)
      • 
              
    
              Account Resources → Include → Specific account → <your account>. Zone Resources → Include → Specific zone → <your domain>. Save the token value somewhere — Cloudflare won’t show it again.
            
    3. 

            
  3. Find your Account ID and Zone ID.
              
      
                
    • Account ID: visible at the top of any Account-level page (Account Home, Workers & Pages sidebar). Or copy it out of the URL: dash.cloudflare.com/<account-id>/...
      • 
                
    • Zone ID: bottom-right of the Overview page for the specific domain.
      • 
              
    
            
    4. 

            
  4. Save the four values in the appliance. Open Configuration → Network tab and:
              
      
                
    • Toggle Cloudflare Tunnel → Enabled.
      • 
                
    • Paste the API token, account ID, and zone ID into the fields that appear.
      • 
                
    • Leave Cloudflare tunnel name at vibe-appliance unless you’re running multiple appliances under one Cloudflare account.
      • 
                
    • Click Save. The values land in /opt/vibe/env/appliance.env (mode 600, root-owned).
      • 
              
    
            
    5. 

            
  5. Run the setup script on the host. SSH in, then:
              
    sudo bash /opt/vibe/appliance/infra/cloudflared-up.sh
    
              The script is idempotent — safe to re-run. It does, in order:
              
      
                
    1. Validates your API token via a no-side-effect account fetch.
      2. 
                
    2. Looks up the tunnel by name; creates it if missing.
      3. 
                
    3. Builds the ingress config from state.apps + manifests — one rule per enabled app’s subdomain plus apex, www, and the infra subdomains (cockpit, portainer, backup).
      4. 
                
    4. PUTs the ingress to Cloudflare; creates or updates one CNAME per published host pointing at <tunnel-id>.cfargotunnel.com.
      5. 
                
    5. Fetches the connector token and writes it to /opt/vibe/env/shared.env as TUNNEL_TOKEN.
      6. 
                
    6. Runs docker compose ... up -d cloudflared — the tunnel container connects to Cloudflare’s edge over outbound TCP 7844.
      7. 
              
    
            
    6. 

            
  6. Verify from outside your LAN (cellular tether is easiest):
              
    curl -sI https://yourdomain.com/admin
curl -sI https://tb.yourdomain.com/
    
              200 / 302 / 401 responses mean the tunnel is working. Connection refused or timeout means the connector isn’t connected — check the container logs:
              
    sudo docker logs vibe-cloudflared --tail 30
    
            
    7. 

            
  7. (Optional) Tighten the firewall — the public internet no longer needs to reach 80/443:
              
    sudo ufw delete allow 80,443/tcp
    
              Only outbound TCP 7844 needs to work going forward. Skip this if you also want HTTP-01 cert renewal (Caddy still listens on 80/443 internally; the tunnel just stops being the only public ingress).
            
    8. 
          


          
Adding apps later

          
The script reads state.apps[*].enabled fresh each run. To publish a new app: enable it from the Apps tab, then re-run the up-script:

          
sudo bash /opt/vibe/appliance/infra/cloudflared-up.sh

          
The script picks up the new subdomain, creates the CNAME, and pushes the updated ingress config. ~5 second round-trip; no container restart needed.


          
To remove

          
sudo bash /opt/vibe/appliance/infra/cloudflared-down.sh

          
Stops the container, deletes only the CNAMEs that point at this tunnel, deletes the tunnel object at Cloudflare, strips TUNNEL_TOKEN from shared.env. Operator-added CNAMEs pointing elsewhere are never touched.


          
Verification

          
    
            
  • sudo docker logs vibe-cloudflared --tail 30 shows lines like “Connection registered with location <X>” — that’s the connector successfully dialing Cloudflare’s edge.
    • 
            
  • Cloudflare dashboard → Zero Trust → Networks → Tunnels: your tunnel shows Healthy with at least one active connector.
    • 
            
  • From outside the LAN, https://yourdomain.com/admin loads, served with a Cloudflare-issued cert.
    • 
          


          
Gotchas

          
    
            
  • Two distinct Cloudflare API tokens. The token you create here (Account.Cloudflare Tunnel:Edit + Zone.DNS:Edit) is for tunnel automation. The separate CLOUDFLARE_API_TOKEN field used for ACME-DNS cert issuance has a narrower scope (Zone.DNS:Edit only). Both can coexist; don’t reuse the same token unless you scope it broadly enough for both jobs.
    • 
            
  • Caddy’s cert situation becomes mostly moot. Cloudflare’s edge does TLS to the real client; the tunnel forwards to https://caddy:443 with noTLSVerify. Caddy still needs some cert to handshake on 443 internally — its self-signed default works fine.
    • 
            
  • Tunnel adds ~50–100ms latency vs a direct connection. Fine for normal browser use; may be noticeable for real-time features.
    • 
            
  • Some apps’ OAuth callback URLs need to be set to the public Cloudflare-served domain. Adjust app-side configs accordingly when migrating from a direct-domain setup.
    • 
            
  • Emergency fallback ports (5171–5198) stay LAN-only by design — we don’t add tunnel ingress rules for them. The whole point of the emergency proxy is to be reachable when the primary path (Cloudflare-Caddy-app) is broken; routing it through Cloudflare too defeats that.
