Help & setup guides

      
Reference for connecting this appliance to the internet via different paths. Pick the one that matches your network situation. Click a title to expand.


      

        Custom DNS — own domain via public DNS
        

          
When to use: Your appliance has a public IP, and you own a domain. Standard public-internet HTTPS via Let’s Encrypt; Caddy auto-issues and renews certs.


          
Prerequisites

          
    
            
  • Domain registered (any registrar)
    • 
            
  • Public-facing host with ports 80 and 443 reachable from the internet
    • 
            
  • An email address for Let’s Encrypt expiry notifications
    • 
          


          
Steps

          
    
            
  1. At your DNS provider, create A records:
              
    yourdomain.com       A   <your-public-ip>
*.yourdomain.com     A   <your-public-ip>
    
            
    2. 
            
  2. Verify propagation from a different machine:
              
    dig +short tb.yourdomain.com
# should return your public IP
    
            
    3. 
            
  3. Open ports 80 and 443:
              
    sudo ufw allow 80,443/tcp
    
              In the DigitalOcean control panel, also confirm any cloud-firewall rules allow 80/443 from 0.0.0.0/0.
            
    4. 
            
  4. Switch the appliance to domain mode:
              
    sudo /opt/vibe/appliance/bootstrap.sh \
  --mode domain \
  --domain yourdomain.com \
  --email you@yourdomain.com
    
            
    5. 
          


          
Verification

          
    
            
  • https://yourdomain.com/admin loads with a green padlock
    • 
            
  • First request to each new subdomain takes 10–30s while Caddy issues its cert — subsequent requests are instant
    • 
          


          
Gotchas

          
    
            
  • Port 80 must be reachable from the public internet for ACME HTTP-01. If port 80 is blocked, certs never issue and every subdomain returns broken — use Cloudflare Tunnel or DNS-01 wildcard certs instead.
    • 
            
  • DNS propagation can take minutes to hours. Don’t switch to domain mode until dig returns the right IP from a fresh machine, or you’ll waste cert-issuance attempts (Let’s Encrypt rate-limits at 5 failures per hostname per hour).
    • 
            
  • Switching from LAN/Tailscale mode rewrites your URLs — update bookmarks and any app callback URLs.
    • 
          

        

      


      

        Tailscale — private mesh, no public IP needed
        

          
When to use: Private deployment — only your team accesses the appliance, no public exposure needed. Or your host has no public IP at all.


          
Prerequisites

          
    
            
  • Tailscale account (free for up to 100 devices)
    • 
            
  • Auth key from your Tailscale admin console
    • 
          


          
Steps

          
    
            
  1. Generate an auth key at https://login.tailscale.com/admin/settings/keys. Recommended: reusable, no expiry — otherwise the appliance will go offline silently when the key expires.
    2. 
            
  2. Switch the appliance to Tailscale mode:
              
    sudo /opt/vibe/appliance/bootstrap.sh \
  --mode tailscale \
  --tailscale-authkey tskey-auth-...
    
            
    3. 
            
  3. Wait a minute or two for the tailnet hostname to propagate. Verify on the host:
              
    tailscale status
    
            
    4. 
          


          
Verification

          
    
            
  • From any tailnet member: https://<hostname>.<your-tailnet>.ts.net/admin loads with a valid Tailscale-CA cert
    • 
            
  • Cockpit: https://<hostname>.<your-tailnet>.ts.net:9090
    • 
          


          
Gotchas

          
    
            
  • Every user accessing the appliance needs Tailscale on their own device.
    • 
            
  • Tailscale’s MagicDNS auto-issues certs from Tailscale’s own CA — no Let’s Encrypt, no port 80 requirement.
    • 
            
  • Vibe-Connect’s client-portal magic-links won’t work if your CLIENTS aren’t on Tailscale (they almost certainly aren’t). For external client traffic, you need Custom DNS or Cloudflare Tunnel; Tailscale is staff-only.
    • 
          

        

      


      

        Cloudflare Tunnel — no port forwarding, no public IP
        

          
When to use: Public clients need to reach the appliance, but you can’t (or won’t) open ports 80/443. Common cases: home installs behind double-NAT or carrier-grade NAT, datacenter VMs on internal networks, or adding Cloudflare’s DDoS protection without changing infrastructure.


          
Prerequisites

          
    
            
  • Cloudflare account (free)
    • 
            
  • Your domain hosted at Cloudflare (transfer nameservers if not already there)
    • 
            
  • Outbound HTTPS allowed from the host
    • 
          


          
Steps — manual; the appliance doesn’t yet automate this

          
    
            
  1. Install cloudflared on the host:
              
    curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg | \
  sudo tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null
echo "deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] \
  https://pkg.cloudflare.com/cloudflared $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/cloudflared.list
sudo apt-get update && sudo apt-get install -y cloudflared
    
            
    2. 
            
  2. Authenticate (opens a browser at Cloudflare):
              
    sudo cloudflared tunnel login
    
            
    3. 
            
  3. Create the tunnel and note its UUID:
              
    sudo cloudflared tunnel create vibe-appliance
    
              Credentials are written to /root/.cloudflared/<UUID>.json.
            
    4. 
            
  4. Create /etc/cloudflared/config.yml (replace <UUID> and yourdomain.com):
              
    tunnel: <UUID>
credentials-file: /root/.cloudflared/<UUID>.json
ingress:
  - hostname: yourdomain.com
    service: http://localhost:80
  - hostname: "*.yourdomain.com"
    service: http://localhost:80
  - service: http_status:404
    
              Cloudflare terminates TLS at its edge with its own cert; local Caddy serves HTTP-only inside the tunnel. Faster, and removes the cert-renewal dependency.
            
    5. 
            
  5. Route the DNS records at Cloudflare:
              
    sudo cloudflared tunnel route dns vibe-appliance yourdomain.com
sudo cloudflared tunnel route dns vibe-appliance "*.yourdomain.com"
    
            
    6. 
            
  6. Run as a systemd service:
              
    sudo cloudflared service install
sudo systemctl start cloudflared
    
            
    7. 
            
  7. Bootstrap the appliance in domain mode — Caddy serves HTTP locally, Cloudflare provides HTTPS:
              
    sudo /opt/vibe/appliance/bootstrap.sh \
  --mode domain \
  --domain yourdomain.com \
  --email noreply@yourdomain.com
    
              The email isn’t used (no Let’s Encrypt) but bootstrap requires the flag.
            
    8. 
            
  8. Tighten the host firewall — ports 80/443 no longer need to be open to the public internet:
              
    sudo ufw delete allow 80,443/tcp
# only outbound to Cloudflare needs to work; inbound is gone
    
            
    9. 
          


          
Verification

          
    
            
  • sudo cloudflared tunnel info vibe-appliance shows status HEALTHY with at least one active connection
    • 
            
  • https://yourdomain.com/admin loads from anywhere on the internet, served by Cloudflare’s cert
    • 
            
  • The Cloudflare Zero Trust dashboard shows the tunnel as Active
    • 
          


          
Gotchas

          
    
            
  • Tunnel adds ~50–100ms latency vs direct connection. Fine for normal browser use; may be noticeable for real-time features.
    • 
            
  • Some apps’ OAuth callback URLs need to be set to the public Cloudflare-served domain.
    • 
            
  • Free tier has no daily/monthly limits for single-app use; rate limits kick in at high traffic. The Cloudflare docs at developers.cloudflare.com/cloudflare-one have the current numbers.
    • 
            
  • If you also want the Vibe emergency fallback ports (5171–5198) public-internet-reachable, you’ll need to add per-port ingress rules to config.yml and route subdomains for them. Most setups leave the fallback ports LAN-only and use Tunnel for the primary domain only — that’s safer.
